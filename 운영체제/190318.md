# 스케쥴링
- 가정
  1. 모든 작업은 같은 시간을 소모한다
  2. 모든 작업은 같은 시간에 도착한다
  3. 모든 작업은 CPU만 사용한다
  4. 실행시간은 알 수 있다
## 퍼포먼스 측정
### Turnaround Time(반환시간) : 완료시간 - 도착시간
- First In, First Out(FIFO 원칙) : 먼저온거먼저
- 먼저 온 작업의 실행시간이 긴 경우 뒤에 금방 끝나는 작업까지 늦어져 비효율적이게 됨
- Shortest Job First(SJF 원칙) : 작업시간이 짧은거 우선 실행
  -> Non-Preemptive(비선점) 스케쥴러의 경우 먼저 온 작업이 시간이 길어버리면 순서를 못 바꿈
  -> Preemptive(선점) 스케쥴러는 새로운 작업이 들어왔을때 현재 작업의 남은시간과 비교해서 순서를 바꿀 수 있음
    -> Shortest Time-to-Completion First(STCF)
### Response Time(반응시간) : 첫 실행시간 - 도착시간
- FPS같은 반응속도가 중요한 게임에서 중요
- Time Slicing : 시간을 나눠서 작업흐름을 계속 옮겨감 (timer-interrupt 기준)
- Fair 한 방법, turnaround time은 느려짐
- time slice가 많아지면 반응시간은 빨라지지만, 그만큼 context switching이 많이 일어나 cache memory등의 작업에 시간을 뺐기는 경우가 있음
  -> tradeoff, 보통 overhead(원래 목적과는 다르게 소모되는 시간이) 1%가 되도록 함
## I/O 포함 작업
- 작업에 I/O 작업이 포함 된 경우, 해당 작업의 I/O쪽으로 흐름이 넘어갔을 때 CPU가 하는게 없음 -> 이 때 다른 작업 수행 가능
