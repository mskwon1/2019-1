# ERD -> Table
## 1 : Entity Relation
- 정규 엔티티 타입은 릴레이션으로 생성
- 키 속성을 그대로 유지
- **복합 애트리뷰트** 는 단순 애트리뷰트만 유지
- 컴포넌트별로 검색을 할거면 따로 릴레이션을 만들고, 아니면 모두 다 릴레이션에 포함??
## 2 : Weak Entity Relation
- 약한 엔티티 타입은 릴레이션으로 생성
- 강한 엔티티의 키 속성을 외래키에 포함시킴
- 기본 키 = 외래 키 + 자신의 부분 키
## 3 : 1:1 관계타입 R
- 별도의 릴레이션을 생성하지 않고, 외래키를 추가
- 한쪽의 기본 키를 선택해서 다른 쪽 릴레이션의 외래키로 추가(기본키 X)
- R에 속한 애트리뷰트는 외래키가 들어간 쪽으로 다 넣어줌
- 두 엔티티 타입 중 전체/완전 참여가 있는 쪽/투플의 개수가 현저히 작은 쪽으로 외래키를 두는게 좋음(null값의 최소화)
## 4 : 외래 키 속성
- 1:N 관계에서는 외래키를 n쪽에 추가
- 모든 1쪽의 애트리뷰트는 N쪽으로 옮김
- 3,4번에서 R의 애트리뷰트가 너무 많으면 따로 릴레이션 생성도 가능
## 5. M:N Relationship Relation
- 지닌 n:m 타입에서는 양쪽의 키 애트리뷰트를 외래키+기본키로 R의 릴레이션에 추가
- R의 애트리뷰트 모두 추가
- 기본키 = 외래키 + 외래키 + 기본키(있으면)
- Logical level에서 두개의 1:N 관계로 표현
## 6. N-ary relationship relation
- 별도의 릴레이션으로 생성
- R의 애트리뷰트 모두 추가
- 참여 엔티티 타입들의 기본키 조합으로 기본키 구성
## 7. Attribute Relation
- 다중값 애트리뷰트는 별도의 릴레이션으로 생성
- 다중값 애트리뷰트와 엔티티 타입의 키 애트리뷰트로 기본키 구성
- 엔티티 타입의 키는 외래키(애트리뷰트가 속해 있던 엔티티)
## Identifying Relationship
- **PK이자 FK인 애트리뷰트가 존재하는지 확인 -> 존재하면 Identifying relationship**
- 5. N : M 관계의 경우
- 2. 존재종속의 경우(Weak entity)
- 7. 다중값 애트리뷰트의 경우
## Weak Entity의 소속
- 그림 상으로 Weak로 그려져있는 경우, 존재종속관계를 잘 파악해야함
- 존재종속관계가 있는 쪽에서만 weak entity로 분류됨(partial key 존재)
- 아닌 쪽에서는 일반 엔티티로 분류됨(primary key가 있어야 함)
## 연속된 존재 종속의 부분 키
- 위 쪽과 마찬가지로, 존재종속 관계를 잘 따져야함
- weak entity가 일반 entity로 취급 받을 때는 키가 있어야 함
## 테이블 구성 예시
- 회원이 회원번호 애트리뷰트가 있고, 하위 개념으로 일반회원/기업회원이 있는 경우
- 테이블 3개를 따로 구성, 회원 테이블에서 주민번호/사업자번호를 FK로 가지고 있음
- 하위 테이블 2개에서는 회원번호는 안가지고 있고, 자신들의 애트리뷰트만 가지고 있음
- **조건 : 두 FK의 존재에 대한 철학차이(sub type 둘다에 속하거나, 둘다 속하지 않는 경우)**
  -> 일반적으로 둘다 허용되지 않음(프로그램이 힘들어짐)
  -> Generalization(Bottom-up 관점) : 레코드는 반드시 sub type중 하나에 속해야 함
  -> Specialization(Top-Down 관점) : sub type에 없는 레코드가 위쪽 타입에 있을 수 있음
